# 設計判断と根拠

判断は「**問い → 検討した代替案 → 採用した案と理由**」の形で記録する。後から読む人が結論だけでなく、なぜ他案を捨てたのかを追えるようにする。

> このファイルはマイルストーン進行に合わせて追記される。骨格段階では各セクションのテーマと書く観点だけ置く。

---

## なぜ ROS 2 から切り離してコアを別リポにするのか

- **検討した代替案**:
  - (A) 既存 `texnitis_move_base_like` パッケージのまま `move_base_flex` に対応する
  - (B) 1 リポに ROS 層と非 ROS 層を共存させ、CMake オプションで切替
  - (C) ★採用：コア（純 C++）と mbf プラグイン（ROS 2）を **別 GitHub リポ** に分割
- **採用理由**: TBD（M1 完了時に書く）。要点として ROS 世代切替への耐性、ユニットテスト容易性、Python / 別ミドルウェアからの再利用、CI のシンプル化を評価予定。

## なぜ pluginlib アダプタを「薄く」するのか

- **検討した代替案**:
  - (A) 既存のように `rclcpp::Node` をプラグイン内部に保持し、パラメータもプラグインで宣言
  - (B) ★採用：プラグインは `Params` 構造体充填と型変換のみ、アルゴリズムはコアに委譲
- **採用理由**: TBD（M3 完了時に書く）。

## なぜ `MapProvider` をシングルトンにするのか

- mbf ros2 ブランチには costmap 入口がないため、各プラグインが /map を購読する必要がある
- 同じ topic を複数プラグインが購読すると帯域・メモリの無駄が発生
- **代替案**:
  - (A) 各プラグインが独自に購読
  - (B) ★採用：`MapProvider::get(node)` で node 単位の唯一インスタンスを共有
- **採用理由**: TBD（M3 完了時に書く）。

## なぜ pybind11 + scikit-build-core なのか

- **検討した代替案**:
  - (A) Cython（学習コスト・C++20 対応の弱さ）
  - (B) ctypes / FFI（型情報が落ちる、numpy 連携が難しい）
  - (C) ★採用：pybind11 + scikit-build-core
- **採用理由**: TBD（M7 完了時に書く）。

## なぜ CasADi はオプション（`NAV_CORE_WITH_MPPI`）なのか

- 組込み Linux など CasADi が入らない環境でも A\* / Pure Pursuit を使えるようにする
- MPPI には CasADi なしのフォールバック実装が既存（旧 `mecanum_mppi_controller`）
- CI では ON / OFF の両方を常に通す
- **採用理由**: TBD（M5 完了時に書く）。

## なぜ Python 側にシミュレータを同梱するのか

- ROS なしで「アルゴリズムが実機相当の条件で動くか」を確認する手段が無かった
- localization 震え・制御遅延・cmd_vel 量子化など、実機でしか拾えない問題を CI で再現する
- **採用理由**: TBD（M8 完了時に書く）。

## なぜ「テスト fixture を最初に作る」のか

- リファクタの最大リスクは「ゴール再到達時に状態管理が壊れる」など、単発テストでは捕まらない振る舞い退行
- 旧コードを正解として固定した上で新コアを書けば、退行を機械的に検出できる
- **採用理由**: TBD（M0 完了時に書く）。
